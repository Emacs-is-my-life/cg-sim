** DONE Fix broken memory release in
CLOSED: [2026-04-01 Wed 21:08]
   =sim/hw/memory/common/memory_region.py=

Current =MemorySpace.release()= has deletion logic nested under the
wrong indentation level:

#+begin_src python
for page_idx_start, mem_region in
    self._regions_by_page_idx_start.items():
    if mem_region.id == free_region.id:
        free_key = page_idx_start
        break

    if free_key is not None:
        self.num_used_pages -=
            self._regions_by_page_idx_start[free_key].num_pages
        del self._regions_by_page_idx_start[free_key]
#+end_src

Because the loop breaks before the delete block runs, releases do
not actually free memory.

Required fix:
- search for the matching region key
- after the loop, if found:
  - decrement =num_used_pages=
  - delete the entry from
    =_regions_by_page_idx_start=

Impact if unfixed:
- memory is never really freed
- false OOM
- false deadlock
- corrupted memory usage accounting

* P0: Compute job state consistency fixes

** DONE Stop mutating region selections inside
CLOSED: [2026-04-01 Wed 21:13]
   =sim/core/job/assertion/compute_assertion.py=

Current compute assertion appends selected regions directly into:

- =job.input_regions=
- =job.output_regions=

This happens during =is_runnable()=, which may be called multiple
times before the job actually begins.

Current pattern:

#+begin_src python
job.input_regions.append(i_mem_region)
job.output_regions.append(o_mem_region)
#+end_src

Required minimum fix:
- clear =job.input_regions= and =job.output_regions= at the start of
  assertion before repopulating them

Better fix:
- make assertion side-effect free
- move exact region binding into begin mutation or another explicit
  reservation step

Impact if unfixed:
- duplicate region entries
- duplicate access count increments
- duplicate decrements
- corrupted access bookkeeping
- possible negative =access_count=

** DONE Fix compute end mutation to release only the regions actually
CLOSED: [2026-04-01 Wed 21:14]
   used by the job

In =sim/core/job/mutation/compute_mutation.py=, begin mutation uses:

- =job.input_regions=
- =job.output_regions=

But end mutation scans by tensor id across all regions in local
memory:

#+begin_src python
candidates = memory.space.get_by_tensor_id(i_tensor_id)
for i_mem_region in candidates:
    i_mem_region.access_count -= 1
#+end_src

Required fix:
- replace tensor-id-wide scans with iteration over:
  - =job.input_regions=
  - =job.output_regions=
- decrement and unlock only those exact regions selected for the job

Impact if unfixed:
- unrelated copies of the same tensor get unlocked
- access counts drift out of sync
- multi-copy placement becomes incorrect

** DONE Redesign compute runnable check so assertion is a pure predicate
CLOSED: [2026-04-01 Wed 21:14]

The file already contains this warning:

#+begin_src python
Warning: assertion and mutation should not be
'system-state-modification' apart.
#+end_src

That warning is correct.

Required design change:
- keep =is_runnable()= read-only
- perform actual reservation / binding atomically in begin mutation
- if reservation fails, do not start the job

Why this matters:
- current correctness depends on assertion and mutation happening
  back-to-back with no interference
- this becomes fragile as queueing or scheduler logic evolves

* P1: Engine and scheduling correctness fixes

** DONE Address head-of-line blocking in
CLOSED: [2026-04-01 Wed 21:19]
   =sim/core/engine/engine.py=

Current waiting-queue handling checks only the front job:

#+begin_src python
job_w = self.job_waiting[0]
if job_w.is_runnable(self.sys):
    ...

    break
#+end_src

Required fix options:

- creates artificial underutilization
- can distort scheduler evaluation results

Impact if unfixed:
- simulator can abort even though executable work exists

* Bottom line

The simulator architecture is promising, but current implementation
still has several correctness landmines:

- immediate engine logging crashes
- broken memory release
- compute job access-state corruption risk
- hardcoded trace expansion behavior
- false deadlock potential from FIFO queue handling
- no usable built-in scheduler yet

This TODO file should be treated as the short-term stabilization plan.
